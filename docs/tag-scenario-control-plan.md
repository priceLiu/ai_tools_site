# 标签 + 场景：可控词表与工作流（2026-05-06 需求与设计）

本文档固定在实施前：**为什么做**、**产品规则**、**与现有清洗的关系**，以及 **开发步骤**。实施进展请同步 [`docs/done.md`](done.md)。

---

## 一、为什么这么重要（标签与场景）

- **场景一级（`tag_categories`，产品文案统称「场景分类」）** 是站内「按用途 / 策展」的主轴之一：首页「按场景找 AI」、`/tag-category/[slug]`、角色页的聚合都以 **标签 → 场景分类** 为纽带。场景的颗粒度可调（大场景 / 小而专），对应不同运营叙事（例如小博主扫全站的浏览路径）。
- **标签（`tags` + `tool_tags`）** 是工具和场景之间的连接器：工具写什么能力、归入哪些场景聚合，主要靠标签关联。提取准、词表可控，场景建设才有坚实基础。
- **左侧产品线菜单（`categories` + `tools.category_id`）** 仍是另一条独立的信息架构（入库栏目 / 导航），与场景体系并行，互不替代。

## 二、需求摘要（本轮实施范围）

### 2.1 提交侧：只允许「自动生成」，禁止自编标签

- 用户 **不得**通过输入框手填新增标签名称。
- 用户 **必须先**点击 **「自动提取标签」**（原「根据介绍自动生成」统一为强调工作流语义），系统在服务端用 **当前库内全部标签**（`tags.name` + `aliases`，并辅以 `TAG_KEYWORDS` 中已有词条的补充关键词）对 **标题（若有） / 概述 / 介绍** 做规则打分，生成建议列表。
- 用户对自动结果 **仅允许删除个别 chip**（过滤明显错误），不允许手加新词；不设静默自动回填（移除 debounce 自动建议），避免用户未觉察就带上标签。
- **新建**：未成功执行过一次「自动提取标签」或结果为空 → **不可提交**。  
  **编辑**：若已有入库标签则可提交；若清空全部标签则需重新提取。

### 2.2 后台：管理员可新建标签与场景分类

- **新建标签**、**六项统计**：在 **`/admin/tag-categories`（场景分类管理）**。名称唯一、必选 **场景分类**，默认勾选 curated。
- **标签清洗**（合并 / 改名 / 删除）：仍在 **`/admin/tags`**。
- **场景分类**：可 **新建分类**、**禁用**（`is_disabled`，前台卡片与聚合页隐身）、在下级 **迁入 / 迁出标签**。
- **脏数据**：沿用现有「待清理」流程；不改变合并逻辑。

### 2.3 性能与缓存

- 全库标签用于匹配的字典通过 **`unstable_cache` + 专用 `revalidateTag`** 缓存；管理员对标签或场景分类的写入一并 **失效字典缓存** 及现有首页 ISR 相关 tag。

### 2.4 顺延（规划中）

- **`is_visible` 独立于禁用**（与 `is_disabled` 并存的产品开关）。
- **两层场景**（组 + 叶子）、首页 **最少展示工具数** 阈值等策略。

上一轮已落地：**`tag_categories.is_disabled`**、 **`/admin/tag-categories`**（统计 + 新建标签 + 场景 CRUD）。

## 四、风险提示

- 全库子串匹配在词量增加时要关注 **误判**（长词包含短词）；后续可加黑名单、阈值或只对「已挂载场景」的标签开放建议（产品开关）。
- 强制「先提取再提交」会多一步交互；需在表单错误提示里说清原因。
