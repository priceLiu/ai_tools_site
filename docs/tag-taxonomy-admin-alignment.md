## 一、原始需求与先前结论（整理）

### 1.1 你对业务的设定（目标语义）

1. **场景分类、角色分类、左侧菜单分类**都可以「聚合一批标签」，也可以与标签在管理上分开维护（松散耦合）。
2. **工具**挂多个标签；可随时清空或重写标签集合。
3. **标签 ↔ 场景、标签 ↔ 角色**都是松散关系（非强制唯一归属）。
4. **场景 / 角色 / 左侧菜单**在语义上是三类不同的「导航聚合维度」，其中左侧菜单**不等价于**「标签的简单别名」，需在模型与文档里**单独说清楚**，不要把三套维度糊成一套。
5. **理想的浏览路径**包括：
   - 场景 → 标签 → 工具  
   - 标签 → 工具  
   - 角色 → 标签 → 工具  
   - 工具 → 标签 → 场景 / 角色（通过详情页的跳转）

### 1.2 先前核对结论（与现状对照）

| 设定项 | 与当前代码/库表是否一致 | 说明 |
|--------|-------------------------|------|
| 场景 ↔ 标签 | **基本一致** | `tags.tag_category_id` → `tag_categories`，可为 NULL；后台可迁入/迁出。 |
| 角色 ↔ 标签 | **基本一致** | `role_category_tags` 多对多；角色页工具列表按联结标签 OR 聚合。 |
| 工具 ↔ 标签 | **一致** | `tool_tags`，最多 20；可清空。 |
| 左侧菜单 ↔ 标签 | **此前不一致（需在计划中补齐）** | 前台 `/category/[slug]` **仅按 `tools.category_id`**（含子树），历史上 **没有**「菜单分类 ↔ 标签」联结表；需在后台能力与文档上与场景管理 **对齐**，并把关系说清楚。 |
| 详情页「工具 → 标签 → 场景/角色」 | **未完成（延后）** | 详情页 `ToolTagsBar` 当前为纯展示，无跳转链接；列为 **待办**，本轮先做前四项对齐与后台能力。 |

### 1.3 本轮硬性约束（你补充的规则）

- **前四点**：产品与模型语义要和上述设定对齐；实施上分阶段勾选。
- **第 5 点（详情路径）**：写入 **「待完成需求」**，本阶段不做验收。
- **第 4 点（左侧菜单）**：要把「菜单分类」和「标签库」的关系 **解耦写清楚**，但 **默认前台列表行为与现在一致**（仍主要靠 `tools.category_id`）；新增的标签联结先做 **管理能力**，是否在 `/category` 列表中与「按 category_id 收录」做 **并集** 或其它策略，在计划中单独一步评审后再改查询（避免无声改变线上口径）。
- **左侧菜单后台**：新增 **与「场景分类管理」同构** 的「菜单分类管理」（禁用、往分类里加标签、移除标签）。
- **预留**：用户 **收藏**「场景分类」「角色分类」（以及后续可扩展菜单分类）；本文件 **仅预留章节**，不纳入当前里程碑验收。

## 二、目标信息架构（对齐后的四条轴线）

```
                    ┌─────────────────┐
                    │   标签库 tags    │
                    │ （全局词表 M:N）  │
                    └────────┬────────┘
         ┌───────────────────┼───────────────────┐
         │                   │                   │
         ▼                   ▼                   ▼
┌────────────────┐ ┌────────────────┐ ┌────────────────────────┐
│ tag_categories │ │ role_categories│ │ categories（菜单/产品线）│
│    （场景）     │ │    （角色）     │ │                        │
└───────┬────────┘ └───────┬────────┘ └───────────┬────────────┘
        │ via               │ via                  │ via（计划）
        │ tag_category_id   │ role_category_tags   │ category_tags（新建）
        │ （已有）          │ （已有）              │
        ▼                   ▼                      ▼
      tool_tags ───────── tool_tags ─────────── tool_tags
        （同一套工具标签边）
```

- **工具主归属（菜单维度）**：`tools.category_id` → `categories`（用于 `/category/...`、面包屑、提交表单主分类），与标签 **正交**。
- **场景维度**：`tag_categories` + `tags.tag_category_id`（与 [`docs/label.md`](./label.md) 一致）。
- **角色维度**：`role_categories` + `role_category_tags`。
- **菜单维度（对齐后）**：`categories` + **`category_tags`（新建）**，表达「该菜单分类在运营上关联哪些标签」；**不改变**「工具挂在哪个菜单分类」仍以 `category_id` 为准，除非你我在后续步骤明确升级列表算法。

## 三、管理后台 — 目标功能清单（验收口径）

完成本轮后应满足：

1. **固定标签库，可持续维护**  
   - 已有：`/admin/tags` 清洗、合并、改名、curated、归属场景等延续使用。
2. **标签库管理**：支持 **新增标签**、**禁用标签**（禁用后前台不再展示该标签页/Chip/联结筛选中的该标签，具体过滤规则在实现阶段写死并在此文档更新）。
3. **场景分类管理**（与现状对齐并文档化）：新建/禁用、向分类迁入标签、从分类移除（现有能力上以文档与字段语义收口）。
4. **角色分类管理**：新建/禁用、联结标签增删（现有能力上以文档收口）。
5. **菜单分类管理**（新增，**交互对齐**「场景分类管理」）：新建或沿用现有 `categories`、**禁用菜单分类**、**category_tags** 联结标签的添加与移除。
6. **清洗能力**：现有合并/改名/删除/一键提取等 **继续使用**，与新字段（如 `tags.is_disabled`）兼容。
