# 微信登录接入指南（网站应用 · PC / 手机浏览器）

> **场景**：用户在 **PC 或手机系统浏览器** 中访问本站，通过 **微信开放平台 → 网站应用** 完成 **扫码或跳转 OAuth** 登录。这是工具站、官网类站点最常见的接入形态。  
> **说明**：本文档为产品与接入准备清单；**具体表单字段、审核周期与命名以 [微信开放平台](https://open.weixin.qq.com) 当前控制台为准**。

---

## 一、与本站现有认证的关系（规划前提）

当前站点为 **邮箱 + 密码** 登录，会话为 **服务端签发 JWT 并写入 HttpOnly Cookie**（如 `setSessionCookie(userId, email, …)`），业务用户标识与 **`profiles.id`** 对齐。

接入微信登录后，技术上应为：**OAuth 成功后仍会落到同一套会话与用户标识**，但在产品上需要先决策：

- 微信用户 **可能没有邮箱**：需在数据库侧增加 **微信标识绑定**（如 `openid` / `unionid` + `profiles.id`），或对会话 payload 做扩展；
- 是否允许 **同一用户绑定邮箱账号与微信**（账号合并 / 解绑策略）。

上述决策需在开发前定稿，再设计表结构与 JWT 字段。

## 二、微信侧：需要申请与准备的事项

### 1. 主体与开放平台账号

- 注册并登录 **微信开放平台**，完成 **开发者资质认证**（个人与企业所需材料不同；企业主体常见需营业执照及对公验证等）。
- 建议提前确定 **接入主体**（哪家公司），避免后续应用归属与资质不一致。

### 2. 创建「网站应用」并开通登录能力

- 在开放平台创建 **网站应用**（非小程序、非公众号）。
- 填写应用信息：**名称、简介、官网 URL** 等。
- 对该应用申请 / 开通 **微信登录** 相关能力（控制台实际名称可能略有差异）。
- 审核通过后获得 **`AppID`**、**`AppSecret`**：**`AppSecret` 仅放在服务端环境变量**，不得写入前端或仓库。

### 3. 域名与合规（常见卡点）

| 项目 | 说明 |
|------|------|
| **授权回调域名 / OAuth 回调 URL** | 在微信后台配置用于接收授权结果的域名（及路径规则）；须为本站可控的稳定 HTTPS 域名。 |
| **域名归属校验** | 开放平台通常要求通过校验文件或 DNS 等方式证明域名所有权。 |
| **隐私政策 / 用户协议** | 官网需提供可访问的隐私政策（及常见配套的条款）；开放平台和审核常会核对站点说明。 |
| **备案** | 若站点面向中国大陆用户且在微信审核语境下要求：**已完成 ICP 备案的域名** 常与审核通过率强相关（以微信平台当期规则为准）。 |

### 4. 环境与运维

- **生产环境必须使用 HTTPS**；回调地址需固定、可在控制台与白名单一致配置。
- 配置 **`SITE_URL` / 公网域名** 等与 OAuth `redirect_uri` 一致，避免拼写与环境分叉错误。

## 三、技术流程概要（OAuth）

以下为典型顺序，便于开发与测试对齐：

1. 用户点击「微信登录」→ 跳转微信授权页（或站内拉起扫码交互，取决于所选产品与前端集成方式）。
2. 用户同意授权 → 微信 **HTTP 302** 回跳到本站 **回调路由**，带回 **`code`**（及 **`state`**，用于防 CSRF）。
3. **服务端**使用 `code` + `AppID` + `AppSecret` 换取 **`access_token`**，并获取 **`openid`**（若应用已绑定开放平台账号，还可能拿到 **`unionid`**，有利于多端统一用户）。
4. 本站根据 `openid` / `unionid` **查找或创建用户**（写 `profiles` 与绑定表），然后调用现有会话逻辑 **写入登录 Cookie**。
5. **`state` 校验**、`code` **一次性消费**、**HTTPS**、**密钥不落前端** 为必须遵守的安全要点。

## 四、与本项目对接时的开发清单（实施阶段）

> 本节在「已通过开放平台审核、已拿到 AppID/AppSecret」之后执行。

1. **数据库**：新增或通过迁移增加 **第三方登录绑定表**（至少存 `user_id`、`provider`、`openid`，可选 `unionid`、绑定时间等）。
2. **API 路由**：  
   - 发起授权（生成 `state`、重定向到微信）；  
   - 回调处理（校验 `state`、换 token、 upsert 用户、**`setSessionCookie`**）。
3. **前端**：登录页增加「微信登录」入口；处理错误态与回跳地址（与现有 `redirect` 查询参数策略对齐）。
4. **会话模型**：若无邮箱，需明确 JWT / `AuthUser` 中 **`email` 的语义**（占位、可空、或扩展字段），并梳理 **用户管理后台**、**管理员重置密码** 等对「仅微信用户」的展示与操作。
5. **可选**：首次微信登录后引导 **绑定邮箱** 或 **设置密码**，与现有 `auth_credentials` 策略统一。
